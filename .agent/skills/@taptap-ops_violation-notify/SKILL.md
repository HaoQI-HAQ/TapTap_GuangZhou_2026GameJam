# Skill: violation-notify

> 帮助 TapTap 开发者理解和应对游戏违规问题。提供违规类型识别、后果说明、整改指引、申诉指导，以及提审前自查排查。

---

## 触发条件

当开发者提到以下内容时触发：
- 收到违规通知/站内信/处罚通知
- 游戏被下架/限制展示/封禁
- 询问某种行为是否违规
- 不知道如何整改/申诉
- 关键词：违规、下架、处罚、整改、申诉、封禁、合规、流量受限
- 想在提审前自查是否有违规风险
- 关键词：自查、合规检查、提审前检查、排查违规

---

## 输入

| 参数 | 类型 | 必需 | 说明 |
|------|------|------|------|
| 违规描述 | string | YES | 开发者描述的情况或收到的通知内容 |
| 游戏名 | string | NO | 涉及的游戏名称 |

---

## 执行步骤

### 步骤 1：识别违规类型

从 8 大类中匹配：

| 大类 | 违规子类型 |
|------|---------|
| 计费违规 | 偷开计费、多次偷开计费 |
| 内容违规 | 色情内容、红色血液、涉政内容、低俗广告 |
| 知识产权 | 素材侵权抄袭、擦边碰瓷热门 IP、游戏 IP 授权存疑 |
| 分发违规 | 引导外部包下载、引导 Q 群下载 |
| 运营违规 | 引导好评、多次更换游戏名、高频使用封闭式测试计划、游戏雷同重复 |
| 资质问题 | 资质异常、没有分发主体、境外无版号、版号运营单位已吊销、防沉迷不合规 |
| 游戏质量 | 游戏不能正常游玩、未屏蔽敏感词、大量低质游戏、盗版抄袭 |
| 账号/厂商级 | 盗版抄袭厂商处罚、低质厂商处罚 |
| **流量受限**（非违规） | 商店页素材不完善导致的展示限制 |

描述模糊时，列出 2-3 个可能类型供开发者确认。

### 步骤 2：说明后果与严重程度

加载 `references/violation-templates.md`，在对应违规类型章节中查找**处罚措施与整改要求**，据此告知开发者：

1. **处罚措施**：从对应子类型的表格中提取处罚内容（如下架、限制展示、不予收录等）
2. **是否有整改机会**：从表格的「整改要求」行确认是否可以整改及具体要求
3. **备注信息**：如表格中有「备注」行，一并告知开发者

**禁止私自编造严重程度分级、整改期限或处罚措施。所有信息必须来自文档原文。**
如果文档中未提及某项信息，明确告知开发者「官方文档中未说明此项，建议联系平台确认」。

### 步骤 3：提供整改指引

从 `references/violation-templates.md` 对应子类型的表格中提取「整改要求」，据此指导开发者。

操作方式：
1. 在对应子类型章节中找到表格的「整改要求」行
2. 提取具体的整改要求（如"删除付费相关内容后重新提审"、"确认并上传授权资质"等）
3. 用开发者易懂的语言转述，但**核心要求必须忠实于文档原文**

**禁止私自编写具体的整改步骤、技术方案（如"接入某某 SDK"）或合规要求。**
如果文档仅给出方向性指引而无具体步骤，如实转达并建议开发者联系平台获取详细指导。

### 步骤 4：申诉指导（如开发者认为误判）

1. 登录开发者中心
2. 找到对应游戏的违规通知
3. 准备申诉材料：原创证明、授权证明、其他支撑材料
4. 通过开发者中心提交申诉
5. 等待平台审核（通常 3-5 个工作日）

### 步骤 5：预防建议

给出预防建议：定期检查内容合规、关注协议更新、提前准备资质、建立内部合规审查流程。

## 特殊情况：流量受限标签

当开发者询问「流量受限」时，**首先明确这不是违规处罚**，而是平台对游戏分发的限制状态。所有通过 Maker 上架的游戏**默认不分发**（即默认流量受限）。

处理流程：
1. 告知开发者这不是违规，不用紧张，Maker 游戏默认都是流量受限状态
2. 说明解除流量受限需要满足的条件：
   - **完善商店页素材**：确保简介、开发者的话、游戏截图、宣传图、实机视频（尤其重要）、首页推荐素材都已上传
   - **论坛发帖申请评审**：前往论坛「游戏上新」板块发帖，帖子需包含游戏链接及游戏分享内容，等待编辑同学评审
   - **风控原因整改**（如适用）：如果是因风控原因受限，需根据风控提供的具体证据和理由进行修改
3. 说明去除后效果：恢复首页推荐流量 + 新品榜入榜资格 + 3 天冷启动流量

详细内容见 `references/violation-templates.md` → 十、流量受限标签

## 自查模式：提审前合规排查

当开发者希望在提审前主动排查违规风险时，使用此模式。

### 触发场景

- 开发者说"帮我自查一下有没有违规"
- 开发者说"提审前检查一下合规"
- 开发者说"我的游戏有没有违规风险"
- 开发者描述了具体的游戏设计，想确认是否合规

### 执行流程

1. **加载自查案例库**：读取 `references/self-check-cases.md`
2. **确认自查范围**：
   - 如果开发者提到了具体领域（如"计费"、"好评"），只展示对应类别的案例
   - 如果开发者要全面自查，展示末尾的「快速自查清单」（23 项）
   - 如果开发者描述了具体行为，直接匹配对应案例给出判断
3. **逐项引导排查**：
   - 展示对应案例的「场景描述」和「自查要点」
   - 引导开发者逐项对照检查
   - 对每项给出「通过 / 存在风险 / 需进一步确认」的判断
4. **汇总结果**：
   - 列出发现的潜在风险项
   - 对每个风险项说明对应的违规类型和可能后果（引用 `violation-templates.md`）
   - 给出整改建议
5. **兜底提醒**：
   - 自查清单基于官方文档已知的违规类型
   - 不能覆盖所有情况，建议同时关注平台最新协议和公告
   - 如有不确定的问题，建议通过开发者中心联系平台确认

### 自查模式输出格式

1. 自查范围 - 本次检查覆盖的类别
2. 逐项检查结果 - 每项的通过/风险状态
3. 风险汇总 - 发现的问题及对应违规类型
4. 整改建议 - 针对每个风险项的处理建议
5. 免责说明 - 自查不能替代平台审核

## 自动扫描：build 前违规检测

每次调用 `build` 工具前，系统会自动运行 `scan.sh` 扫描 `scripts/` 目录中的 Lua 代码，检测潜在违规风险。

### 工作原理

- **触发方式**：通过 Claude Code PreToolUse hook 自动触发，无需手动执行
- **配置位置**：`.claude/settings.json` → hooks → PreToolUse → mcp__sce-urhox__build
- **脚本位置**：`.agent/skills/violation-notify/scan.sh`
- **扫描范围**：`/workspace/scripts/` 下所有 `.lua` 文件

### 检测项目

| 检测类别 | 对应违规类型 | 检测方式 |
|---------|------------|--------|
| 计费相关关键词 | 1.1 偷开计费 | 匹配"充值/内购/付费/支付"等关键词 |
| 外部下载引导 | 4.1 引导外部包下载 | 匹配".apk/.ipa/下载链接"等关键词 |
| 社交群组引导 | 4.2 引导Q群下载 | 匹配"QQ群/群号/加群"等关键词 |
| 好评引导 | 5.1 引导好评 | 匹配"好评/五星/评分送"等关键词 |
| 敏感词过滤缺失 | 7.2 未屏蔽敏感词 | 检测有聊天功能但无敏感词过滤 |
| 外部链接 | 综合检查 | 检测非平台域名的外部 URL |

### 扫描结果

- **无风险**：输出 `[OK] 未发现明显违规风险`，build 正常继续
- **有风险**：输出风险报告（涉及哪些文件、哪些行），build 仍然继续（不阻止构建）

### 误报处理

在代码行末尾添加注释即可忽略该行的检测：

```lua
local text = "充值" -- violation-scan-ignore
```

### 局限性

自动扫描仅覆盖**代码层面可检测的**违规风险，以下仍需人工确认：
- 内容违规（色情/血腥/涉政）— 需检查实际游戏画面
- 知识产权问题 — 需确认素材授权
- 资质问题 — 需在开发者中心确认
---


## 输出格式

1. 违规类型 - 识别出的类型及简要说明
2. 严重程度与后果 - 处罚措施、是否有整改期、期限
3. 整改建议 - 具体可操作的步骤
4. 申诉指引（如适用） - 流程和需准备的材料
5. 预防建议 - 避免再次违规的建议

## 注意事项

### 最重要的规则：绝对禁止私自编写政策内容

**所有关于违规类型、处罚标准、资质要求、运营规范的回答，必须且只能引用 `references/violation-templates.md` 中的原文内容。**

绝对禁止：
- **禁止**私自编写、推测、补充任何跟政策、资质、运营违规相关的内容
- **禁止**通过网络搜索获取违规定义或政策信息
- **禁止**基于通用知识或常识推断违规标准、处罚措施、整改期限
- **禁止**编造文档中不存在的违规类型、处罚措施或合规要求
- **禁止**自行推断某个行为是否违规（文档没写的就是不知道）
- **禁止**给出任何规避违规的建议或变通方案（如"放到游戏外"、"换一种形式"、"用其他渠道"等）。作为平台官方助手，不得引导开发者绕开规则边界

必须做到：
- 回答中涉及的违规定义、处罚标准等，必须能在文档中找到对应原文
- 如果文档中没有对应内容，必须明确告知开发者：「这个问题我目前没有对应的官方说明，建议通过开发者中心联系平台获取准确答复」
- 可以用自己的语言组织回答、引导开发者理解，但**核心事实必须来自文档原文，不得添加文档中没有的信息**

### 其他注意事项

- 语气友好专业：开发者可能焦虑，回复应客观、有建设性
- 优先解决问题：先说怎么改，再说为什么违规
- 不代表平台承诺：申诉结果以平台审核为准
- 引导联系官方：复杂情况建议通过开发者中心联系平台

## 示例

### 输入
> 我的游戏收到站内信说存在偷开计费行为，但我的游戏没有版号，该怎么处理？

### 输出

**违规类型**：计费违规 — 偷开计费

**后果说明**（引用自官方文档）：
根据官方文档，偷开计费的处罚措施为「关闭下载、页面下架、永久不予收录」。

**整改建议**（基于官方文档）：
文档中的整改要求为「删除付费相关内容后重新提审，或提交补充版号资质」。具体而言：
1. 关闭游戏内所有付费入口和充值功能
2. 完成修改后通过开发者中心重新提审

**申诉指引**：
如认为存在误判，可通过开发者中心找到对应违规通知，提交申诉并附上相关证明材料。

**预防建议**：
无版号游戏请确保不包含任何形式的付费功能。如需了解版号申请流程，建议联系平台获取官方指引。
